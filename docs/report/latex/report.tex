% ============================ 1. Requirements ============================
\section{Detailed Requirements and Assumptions}

\subsection{Basic Requirements (Scenario 2)}
The system records information about \textbf{guests}, \textbf{rooms},
\textbf{bookings/reservations}, and \textbf{payments}, as specified by the brief.
Each booking belongs to one guest, can contain one or more rooms, and is
settled by one or more payment transactions.

\subsection{Additional Requirements (Innovation)}
Beyond the basics we introduce five realistic extensions, each grounded in
real hotel-industry operations:

\begin{tabularx}{\linewidth}{@{}p{3.3cm} X@{}}
\toprule
\textbf{Feature} & \textbf{Justification \& Real-world Mapping} \\
\midrule
Loyalty Program       & Returning guests are rewarded with tiered benefits (Bronze\,/\,Silver\,/\,Gold\,/\,Platinum) -- a retention engine standard across Marriott Bonvoy and Hilton Honors. \\
Dynamic Pricing Rule  & Event- and season-based rate adjustment (Christmas, Spring Festival, Summer, weekdays) enables a \emph{revenue-management} layer on top of static base prices. \\
Room Type abstraction & Separates physical room from pricing category, eliminating update anomalies (2NF). \\
Department \& Staff   & Captures hotel organisation; required for any non-trivial PMS. \\
Task Log              & Tracks cleaning, maintenance and inspection workload per (Staff, Room, Task\_Type), enabling KPI analytics. \\
\bottomrule
\end{tabularx}

\subsection{Key Assumptions}
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
    \item A booking can contain multiple rooms across overlapping dates.
    \item Payments may be partial deposits, full settlements, or refunds linked to cancellations.
    \item Loyalty membership is optional; a guest is enrolled at most once (1:1).
    \item Pricing rules are date-windowed and apply globally to all room types; priority breaks overlapping windows.
    \item Concurrent same-room overlap is prevented by application-layer validation; the database stores a \textsc{Check} on date ordering but cannot natively enforce range non-overlap.
\end{itemize}

\section{Conceptual Schema (EER Design)}

The schema comprises 11 entities organised into four functional domains:
\textit{customer}, \textit{inventory}, \textit{transactional}, and
\textit{operational}. The diagram below shows the entity layout, while the
textual model that follows names every relationship explicitly.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\linewidth]{fig1_eer_overview.png}
\caption{EER diagram of the Hotel Booking \& Management System produced via MySQL Workbench Reverse Engineer. Eleven entities span the customer, inventory, transactional and operational domains; \textsc{Booking\_Room\_Detail} is the associative entity resolving the M:N relationship between \textsc{Booking} and \textsc{Room}.}
\label{fig:eer}
\end{figure}

\textbf{Key relationships.}
(i) \textsc{Booking}\,--\,\textsc{Room} is an M:N relationship, decomposed by the
\emph{associative entity} \textsc{Booking\_Room\_Detail} which carries per-room
check-in/out dates and the agreed nightly rate;
(ii) \textsc{Staff}, \textsc{Room}, and \textsc{Task\_Type} form a \emph{ternary}
relationship realised by \textsc{Task\_Log};
(iii) \textsc{Guest}\,--\,\textsc{Loyalty\_Program} is 1:1, optional on the
guest side;
(iv) \textsc{Dynamic\_Pricing\_Rule} is a \emph{global rule entity}, consulted
by application logic rather than via a foreign key -- this allows rules to
evolve independently of the inventory.

\subsection{Normalisation up to 3NF (BCNF check)}
\textbf{1NF.} All attributes are atomic. The draft's monolithic \texttt{LOG} table
conflated booking, room assignment and payment, violating 1NF. It is split
into \textsc{Booking}, \textsc{Booking\_Room\_Detail}, \textsc{Payment}.
\textbf{2NF.} No non-key attribute depends on part of a composite key; the
draft's \texttt{ROOM(roomNo, price, roomType)} is decomposed by abstracting
out \textsc{Room\_Type}.
\textbf{3NF.} No transitive dependencies; \textsc{Payment} stores only
\emph{Booking\_ID}, reaching guest details via
\textsc{Booking}\,$\rightarrow$\,\textsc{Guest}.
\textbf{BCNF.} Every non-trivial FD has a superkey on its left; the schema
is therefore in BCNF.

% ============================ 5. Queries ============================
\section{SQL Queries and Results}

The query set consists of \textbf{12 SQL statements} that collectively reference
\emph{all 11 relations}. They cover multi-table joins, sub-queries
(\texttt{NOT EXISTS}), aggregation with \texttt{HAVING}, window functions
(\texttt{RANK}, \texttt{ROW\_NUMBER}), CTEs, views and a self-join for overlap
detection. The complete script is shipped in \texttt{tests/test\_queries.sql}.

Query Q5 illustrates the database's defensive design. By using
\texttt{NOT~EXISTS} with a date-overlap predicate rather than relying on
\texttt{Room.Current\_Status}, the system correctly handles \emph{future}
reservations on rooms whose live status is still \texttt{Available}.

\begin{lstlisting}[style=sqlstyle,caption={Q5 -- Date-range availability via \texttt{NOT EXISTS}.}]
SELECT r.Room_ID, rt.Category_Name, r.Floor_Level, rt.Base_Nightly_Rate
FROM Room r JOIN Room_Type rt ON r.Type_ID = rt.Type_ID
WHERE r.Current_Status != 'Maintenance'
  AND NOT EXISTS (
    SELECT 1 FROM Booking_Room_Detail brd
    JOIN Booking b ON brd.Booking_ID = b.Booking_ID
    WHERE brd.Room_ID = r.Room_ID
      AND b.Overall_Status NOT IN ('Cancelled')
      AND brd.CheckIn_Date  < '2026-06-05'
      AND brd.CheckOut_Date > '2026-06-01');
\end{lstlisting}

\textbf{Q11 -- VIP scoring} blends total spend, booking frequency and loyalty
tier inside a CTE, producing a defensible 0--100 score:

\begin{lstlisting}[style=sqlstyle,caption={Q11 -- VIP scoring CTE (Guest, Booking, Payment, Loyalty\_Program).}]
WITH customer_metrics AS (
  SELECT g.Guest_ID, COUNT(DISTINCT b.Booking_ID) AS Bookings,
         COALESCE(SUM(p.Amount),0) AS Total_Spent
  FROM Guest g LEFT JOIN Booking b ON g.Guest_ID = b.Guest_ID
       LEFT JOIN Payment p ON b.Booking_ID = p.Booking_ID AND p.Status='Completed'
  GROUP BY g.Guest_ID)
SELECT cm.*, lp.Tier_Level,
  ROUND(LEAST(Total_Spent/10000,1)*40 + LEAST(Bookings/10,1)*30 +
        CASE lp.Tier_Level WHEN 'Platinum' THEN 30 WHEN 'Gold' THEN 22
             WHEN 'Silver' THEN 14 WHEN 'Bronze' THEN 6 ELSE 0 END,1) AS VIP_Score
FROM customer_metrics cm LEFT JOIN Loyalty_Program lp USING (Guest_ID);
\end{lstlisting}

% ============================ 6. Innovation ============================
\section{Innovation: Database-Driven Pricing Agent}

We pair the schema with a Python \emph{Pricing Agent} that consults the
database in real time. For a target date and room type, it reads the
highest-priority active rule from \textsc{Dynamic\_Pricing\_Rule}, computes
the occupancy rate from \textsc{Booking\_Room\_Detail}, and returns
\begin{equation*}
P_\text{final} \;=\; P_\text{base} \;\times\; m_\text{rule} \;\times\; f_\text{occ},
\qquad f_\text{occ} \in \{0.85,\,1.0,\,1.10,\,1.30\}.
\end{equation*}

Figure~\ref{fig:lollipop} summarises the four rules currently loaded. The
Christmas surge ($+80\%$) is the strongest single lever; the year-round
weekday discount ($-15\%$) cushions occupancy in low-demand windows.

\begin{figure}[H]
\centering
\includegraphics[width=0.85\linewidth]{fig6_lollipop_pricing.pdf}
\caption{Four dynamic-pricing rules. Christmas/NY drives the strongest surge ($+80\%$); the weekday discount ($-15\%$) lifts year-round occupancy.}
\label{fig:lollipop}
\end{figure}

Figure~\ref{fig:agentflow} traces the agent's decision flow through the
database. Crucially, the agent's intelligence lives in the data layer:
changing a row in \textsc{Dynamic\_Pricing\_Rule} immediately re-targets
pricing without code redeployment -- exactly the separation a 3NF schema is
built to enable.

\begin{figure}[H]

\caption{Pricing-Agent decision flow. The agent reads rules and occupancy from the database, computes the final price, and returns the recommendation -- a closed loop entirely mediated by SQL.}

\end{figure}

\section{Conclusion}
The system meets all four basic requirements of Scenario~2 and adds five
well-grounded extensions. The schema is normalised to BCNF, exercised by 12
queries spanning every relation, and surfaces actionable insight through
publication-grade visual analytics. The Pricing Agent demonstrates how a
carefully normalised database becomes the substrate for autonomous
decision-making, not merely passive storage.
